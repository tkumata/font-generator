# ADR: Generate Table-Based C And Rust Font Output

## Status

Accepted for Phase 5.

## Context

Phase 5 needs firmware-consumable files from the Phase 4 in-memory font model. The model already contains size-specific glyph metadata and packed 4-bit alpha bitmap bytes.

The output must work for embedded C, Arduino-style C or C++, and embedded Rust without requiring heap allocation.

## Decision

Generate table-based output for both C and Rust.

C output:

- Write `{name}.h` and `{name}.c`.
- Declare glyph metadata and font-size structs in the header.
- Define size-specific bitmap arrays and glyph arrays in the source.
- Define `{name}_sizes` and `{name}_size_count` as the top-level public table.

Rust output:

- Write `{name}.rs`.
- Define `Glyph` and `FontSize` structs.
- Define size-specific bitmap and glyph slices.
- Define `FONT_SIZES` as the top-level public table.

Lookup helpers are deferred. Phase 5 exposes deterministic data tables that firmware examples can scan directly.

## Consequences

Generated output stays simple and deterministic.

Embedded projects can choose their own lookup strategy without needing generated runtime logic.

Phase 6 examples should demonstrate how to locate a glyph from the generated tables.
