# ADR: Use Fontdue For Phase 4 Rasterization

## Status

Accepted for Phase 4.

## Context

Phase 4 needs TTF and OTF font loading, glyph lookup, per-size rasterization, metrics capture, and 8-bit coverage values that can be converted to 4-bit alpha.

The Phase 3 character collector stores Unicode grapheme clusters as display units. A rasterization library still needs a concrete glyph mapping rule for those display units.

## Decision

Use the `fontdue` crate for Phase 4 font loading and grayscale rasterization.

Phase 4 rendering rules:

- Load the configured font path with `fontdue`.
- Treat single-scalar display units as rasterizable candidates.
- Treat multi-scalar grapheme clusters, including variation-selector emoji clusters, as unsupported clusters for now.
- Report unsupported clusters and missing font glyphs through the configured missing glyph policy.
- With `missing_glyphs = "error"`, stop before building output for missing or unsupported units.
- With `missing_glyphs = "skip"`, omit missing or unsupported units and keep them in the in-memory model report.
- Convert `fontdue` 8-bit coverage values to 4-bit alpha using rounding.
- Pack two 4-bit alpha pixels into each byte.

## Consequences

Phase 4 can produce deterministic in-memory glyph data and metrics for scalar glyphs.

Emoji sequences that require shaping, color glyph layers, or variation selector handling are explicitly reported instead of being silently split or substituted.

Full grapheme shaping and color emoji handling remain outside the MVP unless a later phase adds a shaping-capable rasterization path.
