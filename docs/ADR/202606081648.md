# ADR 202606081648: Tight C Fixed Glyph Cells

## Status

Accepted for Phase 10 implementation.

## Context

The fixed C output is intended for LCD firmware that draws generated bitmap records directly. The existing fixed-cell projection can fill the full configured width and height. When firmware draws adjacent records without additional spacing, glyphs can visually touch.

TouchKBD-style firmware also benefits from ASCII records that are half the width of Japanese records. The generated C data should be close to directly usable after copying the header into embedded C code.

## Decision

For `c-fixed` output, treat the configured size as the requested cell size and derive tighter generated dimensions:

- Generated height is `fixed_cell.height - 1`.
- Full-width generated width is `fixed_cell.width - 1`.
- Half-width generated width is `fixed_cell.width / 2 - 1`.
- ASCII letters, digits, punctuation, symbols, and preserved ASCII space are half-width.
- Non-ASCII display units are full-width.

The generated C header will expose full-width and half-width constants, per-character widths, and a fixed two-dimensional bitmap array. The bitmap row stride remains full-width so the data remains index-addressable without heap allocation.

## Consequences

Firmware can draw ASCII and Japanese records with different advances while still selecting data by a stable display-unit index.

Generated rows use the full-width byte count even for half-width records, so half-width data has some zero padding. This trades a small amount of storage for simpler C data and direct indexing.

The change is specific to `c-fixed`; metrics-based C output and Rust output keep the existing model-oriented dimensions.
