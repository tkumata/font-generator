# ADR: Add Rendererless C Fixed Bitmap Output

## Status

Proposed for Phase 8.

## Context

The current C output exposes a metrics-based font model: glyph keys, bitmap offsets, variable bitmap lengths, variable glyph dimensions, advance, and bearings.

That format is useful only when the target firmware already has renderer-like code. Typical microcontroller display firmware does not have a font renderer or text layout engine. It usually needs fixed-size bitmap records that can be drawn with a small loop.

The project goal is compact microcontroller font data, not a generic font model for proportional layout.

## Decision

Add a rendererless C fixed bitmap output format.

The fixed format will generate:

- One fixed-width and fixed-height cell per display unit.
- One fixed-length packed bitmap record per display unit.
- A UTF-8 character mapping whose index matches the bitmap record index.
- Compile-time constants for width, height, bits per pixel, bytes per character, and character count.
- 4-bit alpha data packed two pixels per byte.

The firmware runtime contract is intentionally small:

- Find the display-unit index.
- Select `bitmap_data[index]`.
- Unpack nibbles.
- Draw pixels through the target display driver.

The firmware runtime must not need glyph metrics, bitmap offsets, variable glyph dimensions, advance values, or bearings.

## Consequences

Positive:

- C output becomes immediately usable in common firmware projects.
- Generated data matches the storage shape expected by simple display code.
- Runtime code can stay deterministic and small.
- The generator absorbs placement complexity once on the host machine.

Negative:

- Fixed-cell output can use more flash than tightly packed variable glyph blobs.
- Poorly chosen cell sizes may waste space or fail generation.
- Proportional layout and baseline precision are intentionally not solved by this format.

## Follow-Up

Phase 8 must update documentation, examples, tests, and C output behavior before implementation is treated as complete.
